## 2026-07-07 - T0.2a CODER FIX: out-of-sync lockfile broke CI
**Context:** First Pages deploy failed at `npm ci`: package-lock.json was missing transitive optional deps (`@emnapi/core`, `@emnapi/runtime`), so `npm ci` (exact-match install) errored. Root-caused and reproduced by PM in a clean Linux env; `npm install --package-lock-only` regenerates a synced lock (adds the emnapi entries) and `npm ci` then passes.
**Decision:** CODER FIX (T0.2a) - regenerate the lockfile in sync with package.json (optional deps included), change nothing else, verify `npm ci` succeeds. Keep `npm ci` in the workflow (correct for reproducible CI). Likely cause: lock generated by an npm run that omitted optional deps. Model: Sonnet (trivial mechanical fix).
**Trigger:** GitHub Actions run on the T0.2 commit.

## 2026-07-10 - FPS capped at 60 (PM-direct) after system-lag report
**Context:** User reported window-dragging lagging the whole PC while the game ran. Screenshot diagnosis: Phaser rendering at the display's full 144Hz on a mostly-static scene (confirmed by the dev overlay FPS readout), plus leftover idle Node dev-server processes (user cleanup, harmless). Sign polish rounds T2.6d/e/f also landed prior (plank geometry measured, cost line under title).
**Decision:** PM-direct: `fps: { limit: 60 }` in the Phaser config. 60fps is the design target (mid-range phones) and indistinguishable for this game; halves desktop GPU load. The full profiling pass remains T2.8. Lesson: high-refresh desktops render uncapped by default - cap early on canvas games.
**Trigger:** User performance report during play-gate week.

## 2026-07-10 - T2.9 design calls: tutorial rails, sell steps cut, chain 15 -> 10
**Context:** User found a real stuck-state: plant-mixed lets the player spend all coins on sunwheat alone. He also wanted selling out of the tutorial, conditioned on the tutorial being fully locked ("otherwise they get off the rails").
**Decision:** (1) Full rails: while onboarding is active, a single store-side choke point rejects every action the current step does not call for (plant/harvest/sell/fulfill/skip/expand), and the UI dims or ignores what the store would reject (seed bar selection gating, bag button inert, skip buttons dimmed, orders button only during board steps). Blocked actions fail silently, matching the existing fail-silent philosophy. (2) Sell/bag steps removed; with ORDER B landing in slot 0 the moment ORDER A is fulfilled, the close-then-reopen board dance is pointless too - chain shrinks 15 -> 10 (deliver -> review-order -> close-orders -> plant-mixed), ORDER B insertion moves to entering review-order. (3) Economy without the +32 sale: ORDER A coin reward 63 -> 95, so the 88-coin final planting still fits (ends at 7 coins). XP untouched (30 = exactly level 2 at fulfill). (4) plant-mixed planting is capped per crop (9th sunwheat is a rejected no-op), which with rails makes the "Grow more Sunwheat" fallback dead code - removed. (5) Schema v8: an incomplete save with step > 0 is marked completed (old step indices are meaningless against the new chain, and rails would wedge a mid-chain save whose state has diverged from the scripted walk); step 0 saves keep the tutorial. Model: Fable5/Opus (cross-cutting store+UI gating, precedent-setting choke point). Accepted quirk: sunwheat ripening during plant-mixed cannot be harvested until the step completes - harmless for ~30s, no exception added.
**Trigger:** User day-3 notes + PM economy-walk check.

## 2026-07-08 - T1.9e (PM-direct): swipe phantom-stroke root cause + glow re-dim
**Context:** After T1.9d the user still saw a wrong initial stroke (top-left to top-right) and glow still too hot. Root causes found in PM review: (1) the swipe's perpetual tween is mid-loop whenever a step begins, so the first demo starts at a random path point; (2) trail dots used WRAPPED playhead math, stringing them across the path END (top corner) while the lead sat at the start (right corner) - the eye connects them into a phantom stroke. The path itself was correct.
**Decision:** PM-direct fix: swipe tween restarts from t=0 on every step change AND every reveal transition (post-interruption included); trail playhead clamped instead of wrapped (trail bunches at the stroke start like a finger pressing down); alphas cut to ~a third of the original (halo 0.06-0.16; lead 0.28, glow 0.10, trail 0.14-0.05) - ADD-blend gold over bright green runs hot, so dimming targets perceived brightness, not linear alpha. Lesson: perpetual shared tweens are right for idle breathing but wrong for demonstrations - anything that TEACHES must start at its beginning.
**Trigger:** T1.9d user playtest.

**Addendum (path direction resolved visually):** Verbal iso directions failed twice ("down/left/up" reads differently per viewer), so the PM rendered three candidate paths as diagrams and the user picked OPTION A: start at the TOP corner plot (index 0), row 0 along the upper-right edge to the right corner, serpentine back and forth descending to the bottom - which the user then confirmed was the ORIGINAL T1.9c path all along. The perceived wrong direction was entirely the phantom-trail artifact (wrapped trail + mid-loop tween start); T1.9d's column-major rebuild chased that symptom. Final state: Option A restored + both root causes fixed (clamped trail, restart-on-reveal). Index 0 is also the tap steps' default first-match highlight, so tap target and swipe start coincide naturally. Lessons: (1) for spatial/visual disagreements, show pictures and let the user point; (2) when a report says "X goes the wrong way," verify the mechanism before respecifying the design - the geometry was right and the animation was lying.
