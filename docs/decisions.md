## 2026-07-14 - T3.20 user test FAILED -> T3.20a: background autosaves erase the foreground gap (PM design flaw)
**Context:** Owner's tab-switch test: no WYWA panel after a qualifying 2.5-minute absence. Root cause is the PM's T3.20 design, not the implementation: the foreground summary measured the gap from lastSavedAt, but the 10s autosave interval keeps firing in a backgrounded tab (browser-throttled to ~1/min, still alive), and each background save re-stamps lastSavedAt - on return the measured gap is always under the 120s threshold and matured crops fall outside the window. Unit tests passed because they fake the gap by writing lastSavedAt directly; a real browser has live timers. Deterministic failure for any wait over ~2 minutes.
**Decision:** T3.20a (Sonnet, follow-up): the store records the hidden moment explicitly - handleBackgrounded(atMs = Date.now()) stamps an in-memory hiddenAt (test-injectable timestamp, same spirit as injectable rng/storage) + saves; handleForegroundReturn() computes the summary against hiddenAt (not lastSavedAt), clears it, then rolls the week; computeOfflineSummary gains a sinceMs parameter defaulting to lastSavedAt so the load path is untouched. New regression test: background saves BETWEEN hidden and visible must not erase the gap. LESSON (PM): any "time since the player left" measurement must be anchored by an explicit leave-event timestamp, never by a stamp that live timers keep refreshing.
