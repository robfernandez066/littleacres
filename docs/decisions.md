## 2026-07-07 - T0.2a CODER FIX: out-of-sync lockfile broke CI
**Context:** First Pages deploy failed at `npm ci`: package-lock.json was missing transitive optional deps (`@emnapi/core`, `@emnapi/runtime`), so `npm ci` (exact-match install) errored. Root-caused and reproduced by PM in a clean Linux env; `npm install --package-lock-only` regenerates a synced lock (adds the emnapi entries) and `npm ci` then passes.
**Decision:** CODER FIX (T0.2a) - regenerate the lockfile in sync with package.json (optional deps included), change nothing else, verify `npm ci` succeeds. Keep `npm ci` in the workflow (correct for reproducible CI). Likely cause: lock generated by an npm run that omitted optional deps. Model: Sonnet (trivial mechanical fix).
**Trigger:** GitHub Actions run on the T0.2 commit.

## 2026-07-07 - T1.1 design: derived readiness + provisional economy numbers
**Context:** T1.1 needs plot states (empty -> growing -> ready) and crop economy values. The GDD leaves exact coin numbers as an open question for a future balancing spreadsheet; the roadmap fixes grow times (30s/2m/5m) and unlock levels (1/2/3).
**Decision:** (1) "Ready" is DERIVED from timestamps (now() >= plantedAt + growMs), never stored - only empty/growing exist in the save. Single source of truth makes offline growth free and eliminates ready-flag desync; schema stays at version 1 since the growing variant is additive. (2) Provisional placeholder economy, tunable in config: Sunwheat seedCost 5 / sell 8 / xp 2; Carrot 12 / 20 / 5; Glowberry 30 / 55 / 12. Deliberate shape: short crop has the best coins-per-minute (active play), longer crops better per-plot yield (convenience) - revisit in the balancing pass. (3) Planting spends coins directly (no seed inventory item in MVP; the seeds field stays reserved). (4) Harvest yields 1 crop + the crop's xp; leveling logic waits for T1.7. Model: Fable5/Opus (timestamp/offline math is correctness-critical).
**Trigger:** T1.1 prompt writing.

## 2026-07-07 - T0.5a: lockfile broke CI again; PM fixed directly + CLAUDE.md guard
**Context:** User reported the dev overlay missing on the live site while working on localhost. Diagnosis: deploys #4 (T0.4) and #5 (T0.5) both FAILED at `npm ci` in 13s - the T0.4 vitest install on Windows regenerated package-lock.json and again dropped the Linux-only `@emnapi/*` optional deps (same failure as T0.2a; npm on Windows prunes other platforms' optionals). The live site had been stuck on the T0.3 build, so T0.4 and T0.5 never reached it. PM reproduced the failure and the fix in a clean Linux env against the exact committed files (git show), then (with user approval) wrote the regenerated, npm-ci-verified lockfile directly into the repo rather than a coder round-trip.
**Decision:** PM-direct fix for mechanical, PM-reproducible CI breakage (with user approval) is acceptable; commit T0.5a. Added a standing CLAUDE.md rule: after any dependency change the coder must verify top-level `@emnapi/*` entries survive in the lockfile, regenerating clean if not, and report the check. PM review checklist: whenever a report lists package-lock.json as changed, check for the emnapi entries before COMMIT - this failure is silent locally and only breaks CI. Also: check Actions is green after every push before phone-testing anything.
**Trigger:** User report that overlay worked on localhost but not on the deployed site.

## 2026-07-07 - Model selection per task
**Context:** Coder tasks can run on a stronger model (Fable5/Opus) or a cheaper/faster one (Sonnet). User wants the PM to pick per task rather than default to the strongest.
**Decision:** Every task prompt names a recommended model. Heuristic - use **Fable5/Opus** for tasks with real architectural decisions, subtle correctness (timestamp/offline math, save/migration, GameState design), performance-sensitive systems, or precedent-setting patterns; use **Sonnet** for well-specified, mechanical, low-ambiguity work (config/data entry, UI wiring on an established pattern, straightforward rendering, asset loading, small additive features). When borderline, favor the stronger model for foundational tasks and Sonnet once the pattern exists.
**Trigger:** User request.
